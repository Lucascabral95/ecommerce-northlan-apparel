# Frontend Internationalization

Northlane Apparel uses `next-intl` in the Next.js App Router frontend.

## Locales

- `es` is the default locale.
- `en` is the secondary locale.
- Routes are always prefixed with the locale, for example `/es/products` and `/en/products`.
- Requests without a locale are handled by `apps/web/src/proxy.ts` and redirected to the configured locale.

## Structure

```text
apps/web/src/i18n
  messages
    es.json
    en.json
  metadata.ts
  navigation.ts
  request.ts
  routing.ts
```

Use `@/i18n/navigation` for internal links and router hooks. This keeps navigation locale-aware and avoids hardcoding `/es` or `/en` inside components.

## Adding Text

Add static UI text to both message files under the appropriate namespace:

- `common`
- `navigation`
- `home`
- `products`
- `auth`
- `cart`
- `checkout`
- `account`
- `admin`
- `validation`
- `errors`

Product names, descriptions and SEO fields still come from the API Gateway. Static frontend labels, buttons, empty states and form validation messages should come from the message files.

## Validation

Run the frontend checks after changing translations or localized routes:

```bash
npm run lint --workspace @northlane/web
npm run typecheck --workspace @northlane/web
npm test --workspace @northlane/web
npm run build --workspace @northlane/web
```
